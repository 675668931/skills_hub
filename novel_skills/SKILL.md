# 爽文小说生成器

用于创建、续写、审查和修复长篇爽文项目。核心目标不是“凑够字数”，而是让每章在大纲内推进、与记忆一致、格式可读、结尾不公式化。

## 核心原则

1. **范围优先**：用户指定章节范围时，只处理该范围；删除或忽略范围外临时产物。
2. **先读再写**：续写或修复前必须读取 `output/大纲.md` 与 `.learnings/` 四件套：`STORY_BIBLE.md`、`CHARACTERS.md`、`LOCATIONS.md`、`PLOT_POINTS.md`。
3. **真实情节补字数**：字数不足时补场景动作、战术、对话、代价、环境压力；禁止在章尾追加总结式大段心理/旁白来凑字数。
4. **验证必须落地**：字数、质量、格式和批量模式检查都要运行脚本；不要凭肉眼估算。
5. **记忆必须收敛**：每章完成后更新 `.learnings/`，并把已回收的“进行中”状态改成已完结/已回收，避免后续误导。

## 项目结构

```text
小说名/
├── output/
│   ├── 提示词.md
│   ├── 大纲.md
│   └── 第XX章_章名.md
├── .learnings/
│   ├── STORY_BIBLE.md
│   ├── CHARACTERS.md
│   ├── LOCATIONS.md
│   ├── PLOT_POINTS.md
│   └── ERRORS.md
└── scripts/
    ├── check_chapter_wordcount.py
    └── check_chapter_quality.py
```

初始化新项目：

```bash
${SKILL_DIR}/scripts/init-novel.sh <小说名或目录>
```

## 新项目流程

1. **提示词确认**：用户只给题材时，先补全题材、世界观、主角、金手指、核心冲突、爽点、配角、前三章钩子；未经用户确认，不创建项目文件。
2. **询问总字数**：生成大纲前必须问目标总字数。
3. **生成大纲**：按卷规划，每卷有核心弧线、关键节点、爽点节奏和卷终高潮。
4. **初始化项目**：提示词和小说名确认后运行初始化脚本，写入 `output/提示词.md`、`output/大纲.md`。

## 续写流程

每章开始前写出并执行内部计划，不要把计划写进正文：

```text
章节定位：第XX章，属于第X卷
大纲任务：本章必须完成哪些事件
上章回顾：角色状态、地点、未解决问题
本章冲突：外部压力 + 内部选择/限制
爽点兑现：打脸/突破/救援/反转/压制
偏离风险：可能破坏设定或提前透支战力的点
```

正文结构：

```text
第XX章 章名

【本章概要】一句话
【本章爽点】一句话
【情绪曲线】低开/升级/爆发/钩子
【涉及角色】角色列表
【涉及地点】地点列表

正文：2000-2500字

【章末钩子】一句话
【下章预告】一句话
```

正文要求：

- 每章至少一个明确冲突或爽点。
- 战斗章必须分阶段推进：试探、误判、反制、代价、压制/逆转。
- 对话要体现角色差异；反派误判来自世界观常识，不能降智排队送死。
- 主角可以强，但不能过早满级毁天灭地；升级空间要保留。
- 正文只讲故事，禁止“本章、下一章、后文、剧情、读者、作者、伏笔”等元叙事。
- 禁止 markdown 分隔线 `---`、全角星号 `＊＊`、重复段落、自愈/自动恢复万能机制、未经设定确认的外挂机制。

## 反公式化与可读性硬规则

这是高优先级规则，尤其适用于连续生成五章、十章时。

### 章尾禁止“补丁式尾巴”

不要在正文后 25%-35% 机械追加：

- 连续多段“某角色看着现场，意识到……”式总结。
- 多角色轮流内心感慨，且没有新动作/新信息/新选择。
- “战斗结束 → 众人震惊 → 主角问饭 → 远处黑暗有人愤怒”这种固定尾部骨架连续复用。
- 为补字数硬塞的环境旁白、意义总结、世界观总结。

章尾必须仍然是**场景内推进**：救人、追踪、情报送达、敌方新动作、角色做选择、钩子自然出现。

### 段落与空行

- 正文单行建议 ≤80字，超过必须拆分。
- 正文段落建议 ≤120字，硬上限180字。
- 连续空行不得超过2行；章节中间不得出现大片空白。
- 每5-8段必须有动作、对话、感官或新信息变化，不能连续抽象总结。

### 批量章节结构去重

连续生成多章时，必须在写后比较相邻章节：

- 开头切入方式不能连续复用。
- 结尾钩子类型不能连续复用同一种表达。
- 章尾最后8段不能都由“角色感慨/旁白总结”构成。
- 相邻章节不得复用同一叙述骨架：战场余波→系统结算→众人震惊→主角吐槽→反派摔杯。

如果发现重复，要重写尾部或中段，不要只换词。

## 审查/修复流程

用户说“审查、修复、重写、别再出戏、格式不对”时，按闭环处理：

1. **定位范围**：列出目标章节文件；检查是否有范围外章节或临时文件。
2. **脚本扫描**：运行 `check_chapter_wordcount.py` 和 `check_chapter_quality.py`。
3. **人工模式扫描**：即使脚本通过，也要抽查：长段、空行、章尾公式化、相邻章结构重复、记忆状态过期。
4. **问题清单**：按文件列出具体行号/问题类型。
5. **真实修复**：大段/公式化尾部必须场景级改写；空行和长行可格式化；设定矛盾要改情节而不是只改词。
6. **多轮验证**：修后再次运行脚本；失败继续修。
7. **报告**：只汇报改了哪些文件、哪些问题、脚本结果。

## 必跑命令

在小说项目根目录运行：

```bash
python scripts/check_chapter_wordcount.py --all
python scripts/check_chapter_quality.py --all
```

单章修复时：

```bash
python scripts/check_chapter_wordcount.py output/第XX章_章名.md
python scripts/check_chapter_quality.py output/第XX章_章名.md
```

`check_chapter_quality.py --all` 还会执行跨章批量检查，捕捉连续章节章尾公式化和大段空白。

## 记忆更新规则

每章或每批章节完成后更新：

- 新角色/角色状态 → `CHARACTERS.md`
- 新地点/地点状态 → `LOCATIONS.md`
- 关键事件/伏笔回收 → `PLOT_POINTS.md`
- 世界规则/能力限制 → `STORY_BIBLE.md`
- 失败模式/修复经验 → `ERRORS.md`

过期状态必须清理：例如早期“进行中”“即将开启”“下一阶段”在后续章节完成后要改成“已完结/已回收/持续主线”。

## 失败记录模板

```markdown
## [NOVEL-ERR-YYYYMMDD-XXX] 问题类型

**章节**：第XX章或第XX-YY章
**严重程度**：low | medium | high | critical

### 问题描述
具体问题与行号。

### 根因
为什么生成或审查漏掉。

### 修正方案
如何场景级修复。

### 预防措施
新增脚本检查或写作规则。
```

## 验证清单

- [ ] 章节范围正确，无范围外产物。
- [ ] 每章正文2000-2500字，脚本通过。
- [ ] 质量脚本通过：元叙事、禁用词、长段、单行过长、连续空行、重复段落、批量章尾模式。
- [ ] 章尾不是公式化补字数，最后8段仍有具体动作/信息/选择。
- [ ] 相邻章节开头、收束、钩子没有同骨架复用。
- [ ] `.learnings/` 与最新章节一致，过期“进行中”已清理。
- [ ] 修复报告基于真实脚本输出，不用“应该、估计、看起来”。
